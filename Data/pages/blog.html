Motion Builder Essentials – 01
Characterization & Retargeting

Characterization
Characterization is the process of defining a character’s skeleton in Autodesk MotionBuilder by mapping each bone or joint to MotionBuilder’s standard character template.

Retargeting
Retargeting is the process of transferring animation data from a source character to a target character once both are properly characterized.



Motion Builder Essentials – 02
Poses, Mirroring & Animation Layers

What Are MotionBuilder Poses?

A Pose is a saved snapshot of your character’s skeletal position. It captures transforms of selected joints or the full body so you can quickly reuse the same configuration across animations.

What Can You Do with Poses?

• Save commonly used character positions
• Apply poses to full body or selected parts
• Mirror poses between left and right sides
• Blend poses with existing animation
• Keyframe poses into animation layers
• Maintain non-destructive workflows
• Use Match to keep spatial consistency
• Speed up blocking and pose-to-pose animation
• Build reusable pose libraries
• Improve consistency across shots



Motion Builder Essentials – 03
Motion Builder → Unreal Engine: Exporting Animations

Exporting animations from MotionBuilder to Unreal Engine is a critical step in any game animation pipeline. Even a small mistake during export can result in broken poses, missing motion, or incorrect imports inside Unreal. In this workflow, the key goal is to ensure that all animation data is baked correctly onto the skeleton and exported using the right FBX settings.

Before exporting, it is mandatory to bake the animation to bones. MotionBuilder often drives characters using control rigs or constraints, but Unreal Engine only understands bone animation data. Baking ensures that every frame of animation is converted into keyframes directly on the skeleton, making it fully compatible with Unreal’s animation system.

Key Exporting Steps

• Bake animation to bones before exporting
• Ensure the correct Take is selected for export
• Export only the necessary elements (Bones, Mesh if needed)
• Avoid exporting control rigs, constraints, or extra scene data
• Use FBX export settings optimized for game engines


Baking the animation locks the motion into the skeleton and prevents issues such as missing animation, frozen characters, or incorrect transforms inside Unreal Engine.

FBX Export Settings in MotionBuilder

While exporting, it’s important to review each FBX option carefully. Exporting unnecessary data like cameras, lights, or control rigs can clutter the file and sometimes cause import warnings. Keeping the export clean—focused on bones and animation—results in a stable and predictable import in Unreal Engine. Selecting the correct take ensures that Unreal imports the intended animation clip.

Importing into Unreal Engine

Once inside Unreal Engine, animations should be imported by selecting the correct Skeletal Mesh. Unreal uses the skeletal hierarchy to properly map the animation data. Choosing the wrong skeletal mesh can lead to broken animations, incorrect bone rotations, or failed imports.

After import, it’s always a good practice to preview the animation inside Unreal to confirm:

• The motion plays correctly
• No bones are offset or twisted
• The animation matches what was authored in MotionBuilder


Final Thoughts

A clean export pipeline saves a lot of debugging time later in production. By baking animations to bones, using proper FBX export settings, and selecting the correct skeletal mesh in Unreal Engine, you ensure a reliable and game-ready animation workflow.



Motion Builder Essentials – 04
Blending Animations in Story Mode

Blending Animations in Motion Builder Using Story Mode

In this episode of Motion Builder Essentials, we explore how to blend two animations smoothly using Story Mode in Autodesk MotionBuilder. Instead of cutting directly between clips, we create a seamless transition from a running animation into a sliding animation.

This workflow is especially important in game animation pipelines where smooth transitions directly impact the quality of gameplay.

Why Blending Matters

When two animation clips are placed back-to-back without blending, the result often feels abrupt and unnatural. In action-based animations — such as a run transitioning into a slide — poor transitions can cause visible popping, foot sliding, or unnatural body shifts.

To solve this, Motion Builder’s Story Mode allows us to overlap clips and create proper transition entries.

What This Tutorial Covers

In this tutorial, we:

• Add running and sliding clips into Story Mode
• Overlap both animations to create a transition region
• Use Match Options to align the character properly
• Adjust the transition entry for a smoother blend
• Refine timing for more natural motion


By overlapping the clips, Motion Builder automatically creates a blending zone. Using Match Options ensures that the character’s hips and root position align correctly before the transition occurs.

Using Match Options for Cleaner Transitions

Match Options are a powerful feature when blending animations. They allow you to align position and orientation between clips before the transition begins.

For a run-to-slide example, proper matching helps:

• Maintain forward momentum
• Avoid snapping in the hips
• Reduce foot sliding
• Preserve natural weight shift


Practical Application in Game Development

This type of transition is common in gameplay systems where a player moves from sprinting into a slide or crouch action. Preparing animations properly in Motion Builder ensures they integrate cleanly into engines like Unity or Unreal Engine.

By mastering Story Mode blending and transition entries, you build a stronger foundation for production-ready animation workflows.




Motion Builder Essentials – 05
Multi-Referential Constraint

In this tutorial, I explore the use of the Multi-Referential Constraint in Autodesk MotionBuilder and how it can be used to switch control between different objects during an animation.

For this example, I created a short animation where the character takes a phone from the pocket, makes a phone call, and then places the phone back into the pocket. This type of animation requires changing the parent reference of the prop during the timeline, which is where the Multi-Referential Constraint becomes very useful.

At the start of the animation, the phone is attached to a holder inside the pocket.
The holder reference weight is set to 100, so the phone follows the pocket.

When the character reaches for the phone, I switch the weight from the holder to the Right Wrist Effector by setting the wrist effector weight to 100. This makes the phone follow the hand instead of the pocket.

After the character finishes the phone call, the weight is switched back to the holder so the phone can return to the pocket correctly.

In this setup, I also used a Null element for the phone.
Instead of constraining the animation directly to the phone model, the constraint is applied to a Null object, and the phone is parented to that Null.

This makes the setup more flexible, because if the phone asset needs to be replaced later, the animation will still work without any changes. The animation depends on the Null element rather than the model itself, which is a common workflow in production pipelines.

This workflow is commonly used in MotionBuilder for:

• Prop interaction animations
• Switching object parents during animation
• Motion capture cleanup
• Cinematic animation workflows
• Game animation pipelines
• Asset replacement without breaking animation


The Multi-Referential Constraint is a powerful tool in MotionBuilder and is essential when working with props, especially when an object needs to change its reference during the animation.
